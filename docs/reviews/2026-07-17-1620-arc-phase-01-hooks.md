## Adversarial finding (the core ask): the security fix cannot introduce a false-allow

The reviewer traced whether any crafted payload can make `arc_hook_field` yield
non-empty-but-wrong data that defeats the destructive guard. It cannot: jq/python return
`.tool_input.command` verbatim (no transform makes `rm -rf /` non-destructive), the stub
sanitizer only ever *blanks* the value (forcing the caller's raw-payload superset scan), and
the jq/python keys are literal. Every degraded path (stub output, no-jq, non-JSON) collapses
to the raw scan — proven by the pinned regression (test 10, `noise rm -rf / noise` → exit 2).
The destructive regex is byte-identical to `main`, so its line-scan blind spots are
pre-existing, not introduced. Exit-code logic sound; temp file always cleaned even on early break.
